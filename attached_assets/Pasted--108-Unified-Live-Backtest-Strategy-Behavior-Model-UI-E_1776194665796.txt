#108 — Unified Live/Backtest Strategy Behavior Model, UI Exposure, and Trade/Signal Export

## What & Why
We must stop building separate “live logic” and “backtest logic”.

From this task onward there must be ONE trading system:
- Live = same system running on live data
- Backtest = same system running on historical data

The only difference is the data source and whether orders are simulated or sent.

This task builds the next major layer:
- a strategy behavior model for all 4 active symbols
- exposed in the UI
- driven by the exact live system logic
- usable on historical data through the same runtime path
- with JSON/Excel exports for trades and signals for diagnosis

This is not optional architecture cleanup. This is the foundation that prevents live/backtest divergence forever.

---

## NON-NEGOTIABLE CORE PRINCIPLES

### 1. ONE SYSTEM ONLY
There must not be:
- one live signal path
- one separate backtest signal path
- one separate “simulation logic” path with different exits

Instead:
- one detection path
- one scoring path
- one allocator / trade setup path
- one trade management path
- one behavior-model path

Backtest must run the exact same decision and management code as live, but against historical candles/ticks.

### 2. NO HARDCODED BACKTEST-SPECIFIC TRADE MANAGEMENT
Do NOT keep a backtest-only:
- leg1 logic
- hard SL logic
- MFE exit logic
- synthetic TP/SL path
if live uses something else.

Backtest must use the same live trade-management model.

### 3. STRATEGY BEHAVIOR MUST BE DATA-DERIVED
Do NOT hardcode fixed assumptions for:
- rolling memory window
- scan cadence
- hold duration
- invalidation timing
- expected move profile
- extension probability
- trailing behavior

These must be learned from historical behavior for each:
- symbol
- strategy

### 4. UI EXPOSURE IS REQUIRED
This task must be visible and usable in the app.
Do NOT build it backend-only.

### 5. EXPORT IS REQUIRED
We need clean exports of:
- trades
- signals
- behavior-model outputs
for debugging and analysis.

### 6. DO NOT BREAK LIVE
Current live system must keep running.
Refactor carefully.
If needed, introduce a shared runtime layer and migrate both live and backtest onto it.

---

# OBJECTIVE

Build a unified V3 strategy behavior modeling system for all 4 active symbols:

- BOOM300
- CRASH300
- R_75
- R_100

It must:
1. identify the moves each strategy is supposed to capture
2. profile their historical behavior
3. derive live-useful behavior expectations from that data
4. expose that analysis in the UI
5. run backtests through the exact same live logic path
6. export signals/trades/results for debugging

# PART 2 — STRATEGY BEHAVIOR MODELING

## T005 — Define strategy families per active symbol
For each active symbol, define the expected strategy families:

### BOOM300
- boom expansion / exhaustion continuation behavior

### CRASH300
- crash expansion / reversal-up behavior
- any valid secondary short behavior only if currently live-supported

### R_75
- reversal
- continuation
- breakout

### R_100
- reversal
- continuation
- breakout

These are the moves we expect the system to see and capture.

---

## T006 — Build historical event tagger
Create a strategy event tagger that scans historical data and identifies candidate events for each strategy.

For each tagged event, capture:
- symbol
- strategy
- direction
- entry zone
- start of setup
- actual trigger point
- move development
- failure point if it fails

This is not a live trade yet.
This is the historical truth-tagging layer.

### Important
This tagging must be aligned with the strategy definitions already used in V3.
Do not invent random move classes unrelated to the live engines.

---

## T007 — Build lifecycle profiler for each tagged event
For each historical event tagged above, profile the full lifecycle:

### Before entry
- what pattern formed
- how long it formed for
- what the regime looked like
- what the lead-in shape was
- what the setup-strengthening pattern looked like

### During trade
- MAE path
- MFE path
- time to develop
- time to leg1-equivalent behavior
- time to peak
- pullbacks during move
- whether move was fast / slow / stepped / explosive

### After close / after peak
- whether it continued further
- whether it reversed immediately
- whether it mean-reverted
- whether a trailing logic would have helped
- whether re-entry happened in same direction

This is the behavior model foundation.

---

## T008 — Build per-symbol, per-strategy behavior profile
Using the profiled events, derive a behavior model for each:
- symbol
- strategy
- direction if relevant

At minimum derive:

- expected lead-in duration
- expected trigger window
- expected hold duration
- MAE distribution
- MFE distribution
- time-to-MFE distribution
- time-to-failure distribution
- extension probability after initial target
- typical retracement depth before success
- typical invalidation behavior
- likely scan cadence requirements
- likely rolling memory horizon

### Important
These are not arbitrary config values.
They are learned from historical data.

---

## T009 — Build dynamic strategy expectation model
From the profile above, derive live-usable expectations:

For a live candidate setup, the system must be able to infer:
- what type of move this likely is
- what the expected move envelope is
- how long it should be given to develop
- how much adverse movement is historically normal before success
- when a trade is likely invalid rather than just early
- when a runner is likely vs when leg1 extraction is preferable

This becomes the basis for:
- rolling memory
- scan cadence
- dynamic management
- timeout logic

# PART 3 — ROLLING MEMORY AND STRATEGY-SPECIFIC SCAN DISCIPLINE

## T010 — Derive rolling memory windows from historical behavior
For each symbol-strategy, derive the rolling memory window that should be used in live analysis.

Examples of what this means:
- reversal may need longer structural memory
- continuation may need shorter but stronger trend-state memory
- breakout may need compression + release memory

Do not hardcode one global memory size for all strategies.

### Output required
Store the derived rolling memory horizon per symbol-strategy in a model artifact/config that live can consume.

---

## T011 — Derive scan cadence by strategy
For each symbol-strategy, derive the appropriate live evaluation cadence:
- slower baseline where setup matures slowly
- faster near-trigger cadence where confirmation matters

This must be driven by the historical behavior model, not a global guess.

### Important
Do not let all strategies inherit the same scan cadence if their behavior differs materially.

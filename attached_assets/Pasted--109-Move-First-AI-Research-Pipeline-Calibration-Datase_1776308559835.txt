#109 — Move-First AI Research Pipeline, Calibration Dataset, and Engine/Trade-Management Feeddown

## What & Why
The current “Behavior Model” is answering the wrong question.

Right now the code in:
- `artifacts/api-server/src/routes/behavior.ts`
- `artifacts/api-server/src/core/backtest/behaviorProfiler.ts`
- `artifacts/api-server/src/core/backtest/behaviorCapture.ts`
is profiling the events produced by the current engines/backtest runtime.

That means it is effectively asking:
- “what happens after the engine fires?”

But that is NOT what we want.

We want a move-first calibration system that asks:
- “what are the major, tradeable moves in the historical data?”
- “which of those are breakout / continuation / reversal?”
- “what did the market look like before they started?”
- “how did they behave during the move?”
- “what trade-management pattern would have extracted the most capital?”
- “what engine entry rules capture those moves best with the highest achievable selectivity?”

This task builds that layer.

IMPORTANT:
- AI is RESEARCH ONLY
- AI must NOT run in the live trading loop
- AI must NOT decide live entries/exits in real time
- AI must be used offline / asynchronous / batch-only to enrich calibration data and derive strategy intelligence

This task must repurpose / replace the current behavior-model approach so it becomes:
- move-first
- strategy-first
- profitability-first
- calibration data that feeds DOWN into engines, scoring, trade health, and trade management

---

## CORE OBJECTIVE

Build a multi-pass AI research pipeline that:

1. identifies the major historical moves in the data first
2. labels each move by strategy family:
   - breakout
   - continuation
   - reversal
3. runs multiple AI enrichment passes over each move and its lead-in
4. builds a structured calibration dataset
5. derives the best engine calibration / scoring / trade-management path from that dataset
6. exposes results in the UI
7. exports the results
8. makes it easy to add more symbols later

---

## TARGETS

### Optimization targets
The research system should optimize toward:
- only taking trades where expected extracted move is >= 5%
- maximizing monthly simulated return, targeting >= 50% monthly if the data truly supports it
- maximizing selectivity and move capture, not micro-signal frequency

### Historical-fit target
The system should aim to approach:
- near-100% fit on the HISTORICAL MOVES IT FLAGS AS VALID TARGET MOVES

Important:
- do NOT fake 100%
- do NOT report 100% unless the actual calibration result truly achieves it
- do NOT overfit silently
- always report actual best achieved fit honestly

Meaning:
The AI/calibration loop must keep refining until it either:
- captures all flagged target moves with the strategy definition
OR
- can prove why some target moves are not consistently capturable under one coherent rule set

The goal is:
- find the best achievable historical profitability path
- not generate pretty but fake results

---

## NON-NEGOTIABLE RULES

### 1. AI is research-only
AI must NOT be added to:
- live trade loop
- live entry approval
- live exit approval
- live signal verification

AI is only for:
- offline move labeling
- offline precursor analysis
- offline trade-behavior analysis
- offline calibration
- offline strategy/profile refinement

### 2. Move-first, not signal-first
The pipeline must detect moves first.
It must NOT start by profiling current engine firings.

### 3. Replace or repurpose current Behavior Model
The current Behavior Model is based on replayed engine events.
That logic must not remain the primary calibration model.

You may keep parts of it as:
- downstream validation
- comparison layer
- debug diagnostics

But the new primary model must be move-first.

### 4. No hardcoded TP/SL assumptions
Trade behavior must be learned from the data.
No fixed:
- leg1 percentages
- fixed trailing percentages
- fixed hard stop percentages
- fixed hold windows
unless those are empirically derived and then stored as outputs

### 5. The resulting dataset must feed DOWN into the system
The output of this research pipeline must be usable by:
- engine calibration
- scoring refinement
- trade opening logic
- trade health monitoring
- trade management logic

### 6. Easy to add symbols later
Do NOT make the pipeline hardcoded to only the current 4 symbols.
Build it so additional symbols can be profiled with the same flow later.

# PART 3 — MOVE DETECTION (DETERMINISTIC, CODE-FIRST)

## T003 — Build deterministic move detector
Before AI does anything, code must identify the major historical moves in the dataset.

This move detector must:
- operate on raw/enriched candle data already in the DB
- identify significant tradeable moves
- suppress micro noise
- prefer moves that are:
  - large enough
  - structured enough
  - long enough / clear enough
  - realistically extractable

This move detector must be configurable by:
- symbol
- strategy family
- minimum magnitude
- minimum cleanliness / quality

### Important
This is NOT the live engine.
It is the historical move finder.

The output of this step populates `detected_moves`.

# PART 4 — AI MULTI-PASS RESEARCH PIPELINE

## T006 — Build AI pass runner
Build an asynchronous research job runner that can process moves in batches using GPT-4o API (or the currently configured OpenAI model), outside the trading loop.

The AI runner must:
- split work into multiple passes
- store every pass result in the DB
- support reruns
- support partial reruns
- be resumable
- avoid one giant request that times out

---

## T007 — Pass 1: Precursor discovery
For each detected move, analyze the candles BEFORE the move.

The goal:
- identify recurring precursor structures
- identify setup-building behavior
- identify what differentiates strong moves from false starts
- identify timeframe / duration of buildup
- identify whether precursor pattern matures early, late, suddenly, or gradually

Output should include:
- precursor pattern summary
- setup maturity clues
- candidate feature relationships
- lead-in timing clues

Store results in `move_precursor_passes`.

---

## T008 — Pass 2: Trigger-zone analysis
For each move, identify the best trigger zone.

The goal:
- determine where the move should actually have been entered
- identify how early is too early
- identify what “fully matured setup” looks like
- identify what confirmation should exist at activation time
- determine how much rolling memory is needed to see it properly

This pass should specifically target:
- fewer signals
- higher quality signals
- later but better activation if needed
- avoiding micro re-entries on the same move

Store results in `move_precursor_passes` or a dedicated trigger pass field.

---

## T009 — Pass 3: In-move behavior analysis
For each move, analyze behavior DURING the move.

The goal:
- identify how the move normally develops after entry
- whether it accelerates or steps
- what normal retracements look like
- what adverse movement is normal before success
- how long the move normally takes to mature
- when trailing would help vs hurt
- when pyramiding would help

This pass must produce the clues needed for:
- trade health
- dynamic TP/SL
- dynamic hold logic
- dynamic trailing / protection
- pyramid add logic

Store results in `move_behavior_passes`.

---

## T010 — Pass 4: Best extraction path
For each strategy_family + symbol, compare multiple extraction paths across the detected moves.

The goal:
- determine what trade management extracts the most capital from the targeted moves
- not just highest win rate
- but highest practical profitability path

This pass must test and compare:
- more selective entries
- different entry timing models
- different invalidation models
- different hold-time models
- different runner logic
- different protection logic
- pyramiding suitability

### Important
This is NOT live trading.
This is offline calibration comparison.

The output must estimate:
- expected trade profitability
- expected monthly return
- expected annualized return
- win rate
- average extracted move
- capital efficiency
- drawdown implications

Store aggregated results in `strategy_calibration_profiles`.

---

## T011 — Multi-pass refinement loop
This is critical.

The calibration system must not stop after one pass.

It must keep refining per symbol + strategy until:
- the strategy captures all or nearly all target moves it is supposed to capture
- historical fit is maximized honestly
- profitability path is clearly ranked
- the best calibration profile is identified

### Important
Do NOT fake 100%.
Do NOT force impossible results.
But DO keep iterating until the best achievable fit and profitability path is found and explicitly reported.

The final output must say something like:
- target moves identified
- target moves captured by best calibration
- missed moves and why
- overfit risk
- best profitability path
- recommended live calibration
